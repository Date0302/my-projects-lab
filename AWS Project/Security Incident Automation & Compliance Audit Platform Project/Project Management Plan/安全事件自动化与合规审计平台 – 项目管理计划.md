## 2. 项目目标与范围

### 2.1 项目目标

- 自动化安全事件检测与响应
- 集中管理 AWS 环境及工作负载的安全日志
- 实现合规性持续审计与可视化报告
- 降低安全事件的平均响应时间（MTTR）
- 确保安全策略的一致性与可审计性

### 2.2 项目范围

**包含**：
- 安全检测层：GuardDuty、Inspector、Macie、Config、Control Tower、AWS Organizations
- 日志与事件采集：CloudTrail、VPC Flow Logs
- 数据与事件层：EventBridge、Step Functions、Lambda 编排
- 数据湖与存储：原始日志注入 S3，使用 Athena 查询
- 分析与展示层：QuickSight 仪表板、Athena 查询分析
- 响应层：自动化修复动作（如隔离实例、撤销 IAM 权限）及反馈闭环
- 合规审计：基于 AWS Config 规则 + 自定义 Lambda 评估

**不包含**：
- 第三方 IDS/IPS 集成（可后续扩展）
- 跨 AWS 账号的复杂组织级策略（仅覆盖单账号或有限多账号）
- 本地数据中心日志采集
- SOC 人员培训与工单系统集成

## 4. 假设与约束

### 4.1 假设

- AWS 环境已具备 Organizations 基础结构或可创建
- 安全检测服务（GuardDuty、Inspector 等）可在项目初期启用
- 修复动作仅针对非生产环境先行验证
- 合规框架以 CIS AWS Foundations Benchmark 为初始目标

### 4.2 约束

- 项目为单人实施，无专职安全工程师
- 所有服务需在单一 AWS 区域（如 us-east-1）内部署
- 自动化响应仅限于 AWS 原生服务，不涉及第三方工具
- 成本需控制在 AWS 免费套餐或低预算范围内

## 5. 架构设计概述

基于架构图，平台分为以下层次：

- **管理层**：AWS Organizations、Control Tower 定义多账号治理基线；GuardDuty、Inspector、Macie、Config 提供持续检测与合规评估。
- **检测层**：CloudTrail 记录 API 活动，VPC Flow Logs 记录网络流量。
- **数据与事件层**：所有日志注入 S3 数据湖；EventBridge 捕获事件并触发 Step Functions 或 Lambda 进行编排。
- **响应层**：执行自动化修复动作（如调用 Lambda 隔离资源、发送 SNS 通知），并将结果反馈回系统。
- **分析与展示层**：Athena 对数据湖执行临时查询；QuickSight 构建可视化仪表板，展示安全态势与合规评分。

## 7. 风险管理

| 风险                         | 影响 | 应对措施                                               |
| ---------------------------- | ---- | ------------------------------------------------------ |
| 检测服务误报过多导致响应疲劳 | 中   | 配置 EventBridge 过滤条件，合并同类告警；设置抑制规则  |
| 自动化响应误操作导致业务中断 | 高   | 先对非生产环境启用，增加审批步骤（如 SNS 人工确认）    |
| 日志存储成本失控             | 中   | S3 生命周期策略（热→冷→Glacier），设置 Athena 查询限制 |
| 跨服务权限配置错误           | 高   | IAM 最小权限 + 定期使用 Access Analyzer 审计           |
| 合规规则更新滞后             | 低   | 使用 AWS Config 托管规则 + 每月检查新标准              |

## 8. 成本管理

- **检测服务**：GuardDuty、Inspector、Macie 按用量计费（启用量小的情况下通常较低）
- **存储成本**：S3 分层 + 生命周期（原始日志 30 天后转 IA，90 天后转 Glacier）
- **计算成本**：Lambda 按调用次数计费，Step Functions 按状态转换计费
- **查询成本**：Athena 按扫描数据量计费（建议使用分区和列式格式如 Parquet）
- **可视化**：QuickSight 标准版按用户数收费（单用户免费额度内）
- **优化动作**：设置 AWS Budgets 告警，每月复盘成本报告

## 9. 质量管理

### 9.1 测试策略

| 测试类型 | 范围                                    | 工具/方法                   |
| -------- | --------------------------------------- | --------------------------- |
| 单元测试 | Lambda 响应函数                         | Python unittest             |
| 集成测试 | EventBridge → Step Functions → 修复动作 | 模拟 GuardDuty 事件         |
| 安全测试 | IAM 权限、跨服务访问                    | IAM Policy Simulator        |
| 性能测试 | 高并发事件处理（模拟 50 事件/秒）       | AWS Step Functions 负载测试 |
| 合规验证 | 检查 Config 规则是否按预期评估          | 手动运行合规扫描            |

### 9.2 验收标准

- 关键安全事件（如 IAM 密钥泄露、EC2 恶意行为）在 5 分钟内自动响应
- 所有合规检查项通过率 ≥ 95%
- 响应动作失败时有重试机制并发送告警
- 数据湖中日志保存至少 90 天，且可追溯

## 10. 监控与运维

### 10.1 监控指标

- EventBridge 事件匹配率与延迟
- Step Functions 执行失败率与持续时间
- Lambda 错误率、超时次数
- S3 日志桶大小增长趋势
- CloudWatch 告警：
  - 自动化响应工作流失败次数 > 0
  - Athena 查询扫描数据量超过 1TB/月
  - GuardDuty 产生高危告警

### 10.2 日志与追踪

- 所有 Lambda 日志输出到 CloudWatch Logs
- Step Functions 启用 X-Ray 追踪
- 定期导出 CloudTrail 日志到 S3 用于合规审计

### 10.3 灾难恢复

- 关键配置（IAM、EventBridge 规则、Step Functions 定义）通过 IaC 版本控制
- 数据湖 S3 桶启用跨区域复制（可选）
- 每月执行一次从零部署演练

## 11. 安全策略

- **身份与访问管理**：使用 IAM 角色，遵循最小权限原则；为自动化响应函数分配专用角色
- **数据保护**：S3 桶默认加密（SSE-S3 或 KMS），日志传输使用 TLS
- **事件响应**：所有自动化动作记录到 CloudTrail，形成审计链路
- **合规**：启用 AWS Config 记录所有资源变更，并配置所需规则（如 CIS 基准）
- **隔离边界**：使用 IAM 权限边界限制自动化函数只能操作特定标签的资源
