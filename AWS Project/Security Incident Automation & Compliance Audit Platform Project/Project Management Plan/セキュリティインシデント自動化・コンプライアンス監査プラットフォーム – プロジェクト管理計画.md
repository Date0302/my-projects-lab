## 2. プロジェクト目標と範囲

### 2.1 プロジェクト目標

- セキュリティインシデントの検出と対応を自動化
- AWS環境およびワークロードのセキュリティログを集中管理
- 継続的なコンプライアンス監査と可視化レポートを実現
- セキュリティインシデントの平均応答時間（MTTR）を短縮
- セキュリティポリシーの一貫性と監査可能性を確保

### 2.2 プロジェクト範囲

**対象内**：
- セキュリティ検出層：GuardDuty、Inspector、Macie、Config、Control Tower、AWS Organizations
- ログとイベント収集：CloudTrail、VPC Flow Logs
- データとイベント層：EventBridge、Step Functions、Lambda によるオーケストレーション
- データレイクとストレージ：生ログを S3 に取り込み、Athena でクエリ
- 分析と表示層：QuickSight ダッシュボード、Athena クエリ分析
- 対応層：自動修復アクション（例：インスタンスの隔離、IAM権限の失効）とフィードバックループ
- コンプライアンス監査：AWS Config ルール + カスタム Lambda 評価

**対象外**：
- サードパーティ製 IDS/IPS の統合（後日拡張可能）
- 複雑なマルチアカウント組織ポリシー（単一アカウントまたは限定されたマルチアカウントのみ）
- オンプレミスデータセンターのログ収集
- SOC 担当者トレーニングおよびチケットシステム統合

## 4. 前提条件と制約条件

### 4.1 前提条件

- AWS 環境に Organizations の基盤があるか、作成可能であること
- セキュリティ検出サービス（GuardDuty、Inspector など）はプロジェクト初期に有効化可能
- 修復アクションは非本番環境で先に検証する
- コンプライアンスフレームワークは CIS AWS Foundations Benchmark を初期目標とする

### 4.2 制約条件

- 個人による実装のため、専任のセキュリティエンジニアは不在
- すべてのサービスは単一の AWS リージョン（例：us-east-1）にデプロイする
- 自動応答は AWS ネイティブサービスに限定し、サードパーティツールは使用しない
- コストは AWS 無料枠または低予算内に抑える

## 5. アーキテクチャ概要

アーキテクチャ図に基づき、プラットフォームは以下のレイヤーで構成されます：

- **管理レイヤー**：AWS Organizations、Control Tower がマルチアカウントガバナンスのベースラインを定義；GuardDuty、Inspector、Macie、Config が継続的な検出とコンプライアンス評価を提供。
- **検出レイヤー**：CloudTrail が API アクティビティを記録、VPC Flow Logs がネットワークトラフィックを記録。
- **データ・イベントレイヤー**：すべてのログを S3 データレイクに取り込み；EventBridge がイベントをキャプチャし、Step Functions または Lambda をトリガーしてオーケストレーションを実行。
- **対応レイヤー**：自動修復アクションを実行（例：Lambda を呼び出してリソースを隔離、SNS 通知を送信）し、結果をシステムにフィードバック。
- **分析・表示レイヤー**：Athena がデータレイクに対してアドホッククエリを実行；QuickSight が可視化されたダッシュボードを構築し、セキュリティ状況とコンプライアンススコアを表示。

## 7. リスク管理

| リスク                           | 影響 | 対策                                                         |
| -------------------------------- | ---- | ------------------------------------------------------------ |
| 誤検出の多発によるアラート疲れ   | 中   | EventBridge フィルターの設定、同種アラートの統合、抑制ルールの適用 |
| 自動応答の誤操作による業務中断   | 高   | 非本番環境で先行有効化、承認ステップの追加（例：SNS 手動確認） |
| ログ保存コストの制御不能         | 中   | S3 ライフサイクルポリシー（ホット→コールド→Glacier）、Athena クエリ制限の設定 |
| サービス間の権限設定ミス         | 高   | IAM 最小権限 + 定期的な Access Analyzer 監査                 |
| コンプライアンスルールの更新遅延 | 低   | AWS Config マネージドルールの使用 + 新基準の月次チェック     |

## 8. コスト管理

- **検出サービス**：GuardDuty、Inspector、Macie は使用量に応じた課金（小規模な有効化では通常低額）
- **ストレージコスト**：S3 階層化 + ライフサイクル（生ログは30日後に IA、90日後に Glacier に移行）
- **コンピューティングコスト**：Lambda は呼び出し回数、Step Functions は状態遷移回数で課金
- **クエリコスト**：Athena はスキャンデータ量で課金（パーティションと Parquet などの列形式を推奨）
- **可視化**：QuickSight スタンダード版はユーザー数で課金（単一ユーザーは無料枠内）
- **最適化アクション**：AWS Budgets アラートを設定し、毎月コストレポートをレビュー

## 9. 品質管理

### 9.1 テスト戦略

| テスト種別           | 範囲                                              | ツール/方法                          |
| -------------------- | ------------------------------------------------- | ------------------------------------ |
| 単体テスト           | Lambda 応答関数                                   | Python unittest                      |
| 統合テスト           | EventBridge → Step Functions → 修復アクション     | GuardDuty イベントのシミュレーション |
| セキュリティテスト   | IAM 権限、サービス間アクセス                      | IAM Policy Simulator                 |
| パフォーマンステスト | 高負荷イベント処理（50イベント/秒をシミュレート） | AWS Step Functions 負荷テスト        |
| コンプライアンス検証 | Config ルールが期待通りに評価されるか             | 手動コンプライアンススキャン         |

### 9.2 受入基準

- 重要なセキュリティイベント（例：IAM キー漏洩、EC2 悪意ある動作）が5分以内に自動応答されること
- すべてのコンプライアンスチェック項目の合格率 ≥ 95%
- 修復アクション失敗時に再試行メカニズムとアラートが動作すること
- データレイク内のログが少なくとも90日間保存され、完全なトレーサビリティを確保すること

## 10. 監視と運用

### 10.1 監視メトリクス

- EventBridge イベントマッチ率と遅延
- Step Functions 実行失敗率と実行時間
- Lambda エラー率、タイムアウト回数
- S3 ログバケットの成長傾向
- CloudWatch アラート：
  - 自動応答ワークフローの失敗回数 > 0
  - Athena クエリのスキャンデータ量が月間 1TB を超過
  - GuardDuty が高重要度検出を生成

### 10.2 ロギングとトレーシング

- すべての Lambda ログを CloudWatch Logs に出力
- Step Functions で X-Ray トレーシングを有効化
- コンプライアンス監査のため CloudTrail ログを定期的に S3 にエクスポート

### 10.3 ディザスタリカバリ

- 重要な設定（IAM、EventBridge ルール、Step Functions 定義）は IaC でバージョン管理
- データレイク S3 バケットのクロスリージョンレプリケーション（オプション）
- 毎月のゼロからのデプロイ訓練

## 11. セキュリティポリシー

- **アイデンティティとアクセス管理**：最小権限の原則に従った IAM ロールを使用；自動応答関数に専用ロールを割り当て
- **データ保護**：S3 バケットのデフォルト暗号化（SSE-S3 または KMS）、ログ伝送には TLS を使用
- **インシデント対応**：すべての自動アクションを CloudTrail に記録し、監査証跡を形成
- **コンプライアンス**：AWS Config を有効化してすべてのリソース変更を記録し、必要なルール（CIS ベンチマークなど）を設定
- **分離境界**：IAM パーミッションバウンダリを使用して、自動関数が特定のタグを持つリソースのみ操作できるように制限
