## 2. プロジェクト目標と範囲

### 2.1 プロジェクト目標

- 画像の自動アップロードと処理フローを実現
- システムの拡張性と安定性を向上
- 人的な運用・保守コストを削減
- データアクセスの安全性を確保

### 2.2 プロジェクト範囲

**対象内**：
- 画像アップロードモジュール（API Gateway + Lambda + S3）
- 画像処理モジュール（圧縮、透かし入れ）
- メッセージキューとイベント駆動フロー（SQS による疎結合）
- データ保存とライフサイクル管理（S3 + DynamoDB）
- ユーザー認証とアクセス制御（Cognito + IAM）
- 処理完了通知（SNS）

**対象外**：
- フロントエンドUI（APIのみ提供）
- AI/機械学習による画像認識
- 動画処理
- オンプレミスまたはハイブリッドクラウド展開

## 4. 前提条件と制約条件

### 4.1 前提条件

- ユーザーのネットワークからAWSサービスへ正常にアクセスできること
- 一般的な画像フォーマット（JPEG、PNG）のみを処理
- AWS無料枠または低コストアカウントを使用、予算承認は不要
- Lambdaの最大実行時間15分で通常の画像処理が完了する

### 4.2 制約条件

- EC2やKubernetesは使用せず、サーバーレスサービスのみ
- GPUアクセラレーションは使用しない
- 超大ファイル（>1GB）は非対応
- 個人開発のため、専任のテストチームは不在

## 5. アーキテクチャ概要

- S3：元画像バケット + 処理済み画像バケット
- API Gateway + Lambda：アップロードエンドポイント
- SQS：非同期疎結合、ピーク時の負荷平滑化
- Lambda（処理関数）：圧縮、透かし入れ
- SNS：処理完了通知の送信
- DynamoDB：画像メタデータを保存（ファイル名、サイズ、処理時間、ステータス）
- Cognito：ユーザー認証 + 一時的認証情報
- IAM + セキュリティ境界：最小権限による分離

## 7. リスク管理

| リスク                       | 影響 | 対策                                                    |
| ---------------------------- | ---- | ------------------------------------------------------- |
| 高負荷によるシステム圧力     | 高   | SQSによる負荷平滑化、Lambda同時実行予約                 |
| 権限設定ミスによるデータ漏洩 | 高   | IAM最小権限 + セキュリティ境界 + 定期的な監査           |
| メッセージ処理失敗または損失 | 中   | SQSデッドレターキュー + 再試行メカニズム + 失敗アラーム |
| ストレージコストの過大       | 中   | S3ライフサイクルポリシー（標準 → IA → Glacier）         |
| Lambda実行タイムアウト       | 低   | 画像サイズを制限、または複数ステップに分割              |

## 8. コスト管理

- コンピューティングコスト：Lambdaのみ（呼び出し回数と実行時間で課金）、アイドル時はコストゼロ
- ストレージコスト：S3階層化 + DynamoDBオンデマンド
- 通知コスト：SNS（月間最初の100万リクエストまで無料）
- キューコスト：SQS（月間最初の100万リクエストまで無料）
- 監視：CloudWatch無料枠で十分
- 最適化：AWS Budgetsアラートを設定して予算超過を防止

## 9. 品質管理

### 9.1 テスト戦略

| テスト種別           | 範囲                              | ツール/方法              |
| -------------------- | --------------------------------- | ------------------------ |
| 単体テスト           | Lambda関数                        | Python unittest          |
| 統合テスト           | API → S3 → SQS → Lambda → SNS     | AWS CLI                  |
| セキュリティテスト   | IAM権限、Cognito一時認証情報      | AWS IAM Policy Simulator |
| パフォーマンステスト | 100同時アップロードをシミュレート | Artillery / Locust       |
| 可観測性             | ログ、エラー、遅延                | CloudWatch Logs + X-Ray  |

### 9.2 受入基準

- 画像アップロード後30秒以内に処理完了（99%のリクエスト）
- IAM権限昇格の警告が発生しない
- SQSデッドレターキューが3回失敗後に正しくメッセージを受信する
- すべてのAPIリクエストはCognito認証を必要とする

## 10. 監視と運用

### 10.1 監視メトリクス

- Lambdaの呼び出し回数、エラー率、実行時間
- SQSキューの長さ（可視メッセージ + 不可視メッセージ）
- S3バケットサイズとリクエスト数
- DynamoDBの読み書きキャパシティ使用量
- CloudWatchアラームしきい値：
  - Lambdaエラー率 > 5%
  - SQS滞留 > 100メッセージが5分間継続

### 10.2 ロギングとトレーシング

- すべてのLambdaログをCloudWatch Logsに出力
- 重要なリクエストのトレースにX-Rayを有効化
- 長期監査のために定期的にログをS3にエクスポート

### 10.3 ディザスタリカバリ

- S3のクロスリージョンレプリケーション（オプション、コスト次第）
- DynamoDBのオンデマンドバックアップ（週次）
- メタデータ再構築可能：S3バケットを再スキャンしてDynamoDBレコードを再生成

## 11. セキュリティポリシー

- ユーザー認証：Cognitoユーザープール + IDプール
- 一時的認証情報：Cognito経由で取得、特定のS3プレフィックスに制限
- IAMロール：
  - API呼び出しロール：元S3バケットへの書き込み + SQS送信のみ許可
  - 処理関数ロール：元バケットの読み取り、処理済みバケットへの書き込み、DynamoDB書き込み、SNS公開のみ許可
  - セキュリティ境界：明示的なリソースARNを使用、ワイルドカード`*`は使用しない
- 暗号化：S3サーバーサイド暗号化（SSE-S3）、DynamoDBデフォルト暗号化
