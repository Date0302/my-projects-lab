# README

#### 🌐 Language / 言語 / 语言

🇺🇸English: [README.md](README.md) | 🇯🇵日本語: [README.ja.md](README.ja.md) | 🇨🇳中文: [README.CN.md](README.CN.md)

このプロジェクトは、AWS サーバーレスアーキテクチャ上に構築された、コストを考慮したインテリジェントなドキュメント QA システムです。中核となる革新性は、質問の複雑さに応じて異なるコストの AI モデルに動的にルーティングし、コストをファーストクラスの市民として追跡、分析、可視化する点にあります。これは、機能、パフォーマンス、コスト間のトレードオフにおけるアーキテクトの技術を完全に体現しています。

## コア機能

- 完全サーバーレス：Lambda、API Gateway、S3、DynamoDB などのマネージドサービス上に構築 – 自動スケーリング、サーバー管理ゼロ
- コストを考慮した動的ルーティング：質問の複雑さに基づいて LLM モデル（Titan Lite vs. Claude 3）をインテリジェントに選択、ユーザー体験を損なうことなく推論コストを大幅に削減
- RAG（検索拡張生成）：OpenSearch Serverless を使用したセマンティック検索により、AI が企業のプライベートドキュメントに基づいて質問に回答
- 本番レベルの可観測性：呼び出しごとのモデル、トークン数、コストを完全に追跡、QuickSight で可視化、CloudWatch でアラート
- セキュリティとコンプライアンス：Cognito 認証、S3 暗号化、API Gateway のスロットリングと CORS 制御
- 信頼性設計：Step Functions の再試行と DLQ（デッドレターキュー）、DynamoDB TTL による自動クリーンアップ

## このプロジェクトの理由

ソリューションアーキテクトの候補者として、私はシステムを「構築」できるだけでなく、十分な根拠に基づいたアーキテクチャ上の意思決定を行う必要があります。企業が生成 AI を導入する際の最大の課題の 1 つは、制御不能な推論コストです – すべてのリクエストに対して最も強力なモデル（例：Claude 3）を盲目的に呼び出すとコストが爆発し、一方ですべてのリクエストに低コストモデルを一律に使用すると、複雑な質問の回答品質が犠牲になる可能性があります。

このプロジェクトは、動的ルーティングによってこの矛盾を解決します。システムは質問タイプを自動的に識別し、単純な頻出質問は低コストパスに、複雑な推論質問は高品質パスにルーティングし、コストを可視化します – これにより、ビジネス関係者にデータ駆動型のコスト最適化の推奨事項を提供します。これこそがソリューションアーキテクトの中核的価値です。

## コンポーネント詳細

### 1. データ取り込みパイプライン

| コンポーネント                   | 技術選定                          | 設計判断                                                     |
| -------------------------------- | --------------------------------- | ------------------------------------------------------------ |
| ドキュメントアップロード         | API Gateway + S3（生バケット）    | プリサインド URL パターンを使用し、フロントエンドが S3 に直接書き込むセキュリティリスクを回避 |
| ワークフローオーケストレーション | AWS Step Functions                | 再試行、エラーハンドリング、視覚的デバッグをサポート – 単純な Lambda チェーンよりも堅牢 |
| テキスト処理                     | Lambda（Python + PyPDF2/chardet） | サーバーレス、呼び出しごとの課金、複数のドキュメント形式を自動処理 |
| ベクトル化                       | Amazon Bedrock - Titan Embeddings | 他の Bedrock モデルとの統一エコシステム、低コスト（約 $0.0001/1K トークン） |
| ベクトルストレージ               | Amazon OpenSearch Serverless      | クラスター管理不要、自動スケーリング、ベクトル検索とメタデータフィルタリングをネイティブサポート |

**重要設計**: S3 イベントは Step Functions を直接トリガーできません。EventBridge または Lambda ブリッジを間に使用します（アーキテクチャ図に暗黙的に含まれています）。

### 2. クエリ＆コスト認識パイプライン

これはシステムの中核的な意思決定層です。

| コンポーネント             | 技術選定                                                | 設計判断                                                     |
| -------------------------- | ------------------------------------------------------- | ------------------------------------------------------------ |
| ユーザー認証               | Amazon Cognito                                          | マネージドユーザープール、API Gateway とネイティブ統合、すぐに使える JWT 検証 |
| クエリエントリ             | API Gateway（REST + WebSocket）                         | REST は単一リクエスト用、WebSocket は将来のストリーミングレスポンスのために予約 |
| コアオーケストレーター     | Lambda（Orchestrator）                                  | すべてのビジネスロジックをホスト：検索、複雑さ判断、モデルルーティング、コスト記録 |
| セマンティック検索         | OpenSearch Serverless ベクトル検索                      | 最も関連性の高い Top-K ドキュメントチャンクをコンテキストとして返す |
| モデルルーティングロジック | ルールベース（質問長/キーワード）+ 軽量分類器に拡張可能 | 初期ルール：質問が 20 文字未満、または「とは/定義」などのキーワードを含む → 低コストパス；「なぜ/比較/分析」を含む、または長い質問 → 高能力パス |
| 低コストモデル             | Amazon Bedrock - Titan Text G1‑Lite                     | 高速推論、コスト約 $0.0003/1K 入力トークン、事実ベースの QA に適す |
| 高能力モデル               | Amazon Bedrock - Claude 3 Haiku/Sonnet                  | 強力な推論、複雑なロジックをサポート、コスト約 $0.0025/1K 入力トークン（Titan の約 8 倍） |
| コスト記録                 | Amazon DynamoDB                                         | 高い書き込みスループット、TTL 自動クリーンアップ、呼び出しごとのタイムスタンプ、モデル、トークン数、推定コストを記録 |

**入出力例**（Orchestrator Lambda）：

```json
// 入力（ユーザーの質問）
{
  "userId": "user123",
  "question": "サービスレベル契約（SLA）とは何か説明してください。"
}

// 出力（システム応答）
{
  "answer": "サービスレベル契約（SLA）は、サービスプロバイダーと顧客間の正式な約束です...",
  "model_used": "amazon.titan-text-lite",
  "cost_estimate_usd": 0.00021,
  "latency_ms": 520
}
```

### 3. コスト分析＆可視化パイプライン

| コンポーネント       | 技術選定                              | 設計判断                                                     |
| -------------------- | ------------------------------------- | ------------------------------------------------------------ |
| コストデータレイク   | DynamoDB エクスポート → S3（Parquet） | DynamoDB を時間パーティションでエクスポート、Parquet に変換して Athena による効率的なクエリを実現 |
| サーバーレスクエリ   | Amazon Athena                         | コストログの標準 SQL 分析、クラスターのプロビジョニング不要  |
| 可視化ダッシュボード | Amazon QuickSight                     | マネージド BI ツール、自動リフレッシュと行レベルのセキュリティをサポート |
| 予算アラート         | CloudWatch + SNS                      | 日次累計コストを監視、しきい値（例：$5/日）を超えるとメール/SMS をトリガー |

**ダッシュボードの主要指標**：

- 過去 7 日間 / 30 日間の総コスト推移
- モデル別の呼び出し回数とコスト割合
- 単純な質問と複雑な質問の比率
- リクエストあたりの平均コスト（ビジネスへの ROI 報告用）

## アーキテクチャのトレードオフと設計判断

ソリューションアーキテクトのポートフォリオプロジェクトとして、以下の主要なトレードオフを**明示的に文書化**します：

| 判断ポイント           | オプション A            | オプション B                 | 最終選択               | 正当性                                                       |
| ---------------------- | ----------------------- | ---------------------------- | ---------------------- | ------------------------------------------------------------ |
| モデルルーティング戦略 | 常に Claude 3 を使用    | 動的ルールベースルーティング | 動的ルーティング       | コストを 70-80% 削減、かつ日常的な質問の 80% は低コストモデルで対応可能 |
| ベクトルデータベース   | Aurora PG with pgvector | OpenSearch Serverless        | OpenSearch Serverless  | クラスター管理不要、Bedrock との統合がより自然、リクエストごとの課金 |
| リアルタイム要件       | LLM 応答を同期待ち      | 非同期ポーリング             | 同期                   | QA シナリオはレイテンシーに敏感（目標 <2 秒）、同期は許容可能で実装も簡単 |
| コスト追跡粒度         | 日次集計のみ            | 呼び出しごとの詳細記録       | 呼び出しごとの詳細記録 | その後の最適化（例：最もコストのかかるユーザーや質問タイプの特定）のための細粒度のコスト帰属を提供 |

## 監視と運用

システムは CloudWatch と統合され、包括的な監視を実現しています：

- 各 Lambda 関数の実行ログ、実行時間、エラー率、スロットリング数を表示
- CloudWatch コンソールでカスタムコスト指標（例：`TotalDailyCost` が $10 を超えた場合）に基づくアラートを設定
- Step Functions ワークフローの視覚的トレースによるドキュメント処理状況の追跡
- X-Ray 分散トレーシング（オプション）によるエンドツーエンドのレイテンシーボトルネック分析

## 実行方法（概要）

1. AWS CDK または Terraform を使用してすべてのリソースをワンクリックでデプロイ（IaC テンプレート提供済み）
2. Cognito でテストユーザーを作成し、JWT トークンを取得
3. API Gateway 経由でドキュメントをアップロード（POST /documents）
4. API Gateway 経由で質問を送信（POST /query）、JWT トークンを添付
5. 30 秒待つ（ベクトル化が完了するまで）、再度質問するとドキュメントに基づいた回答が得られる
6. QuickSight でコストダッシュボードを開き、異なるモデルパス間のコスト分布を観察
7. CloudWatch の予算しきい値を設定し、SNS アラートを検証

このプロジェクトはソリューションアーキテクトのポートフォリオプロジェクトとして完了しました。すべてのコード、IaC テンプレート、アーキテクチャ図、デモビデオはリポジトリで入手できます。コストデータは Bedrock のオンデマンド価格に基づく実際の見積もりです。
