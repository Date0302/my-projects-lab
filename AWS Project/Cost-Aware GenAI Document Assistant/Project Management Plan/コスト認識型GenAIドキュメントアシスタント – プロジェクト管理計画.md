## 2. プロジェクト目標と範囲

### 2.1 プロジェクト目標

- 完全サーバーレスで自動スケーリングするドキュメントQAシステムを構築
- 質問の複雑さに基づいた動的モデルルーティング（低コストTitan vs 高能力Claude 3）を実装
- 包括的なコスト追跡と可視化ダッシュボードを提供
- データセキュリティとユーザー認証を確保
- ソリューションアーキテクトのポートフォリオプロジェクトとして、アーキテクチャのトレードオフと意思決定を提示

### 2.2 プロジェクト範囲

**対象内**：
- ドキュメントアップロード・処理パイプライン（API Gateway → S3 → Step Functions → テキスト抽出 → ベクトル化 → OpenSearch）
- クエリ・ルーティングパイプライン（Cognito認証 → API Gateway → Orchestrator Lambda → 意味検索 → モデルルーティング → 回答生成）
- コスト追跡パイプライン（呼び出しごとにモデル、トークン数、推定コストをDynamoDBに記録）
- コスト分析・可視化（DynamoDBエクスポート → Athena → QuickSight）
- 監視とアラート（CloudWatchログ、カスタムコストメトリクス、SNSアラート）
- セキュリティ設計（IAM最小権限、S3暗号化、API Gatewayスロットリング）

**対象外**：
- マルチターン会話のコンテキスト記憶（単発QAのみ）
- リアルタイムストリーミング応答（WebSocketは予約済みだが未実装）
- 複雑なカスタムモデルファインチューニング
- 非テキストファイルのサポート（例：音声・動画）
- プロダクショングレードのマルチリージョン展開

## 4. 前提条件と制約条件

### 4.1 前提条件

- AWSアカウントでBedrockモデルアクセスが有効化済み（Titan、Claude 3）
- ドキュメント形式はPDF、TXT、一般的なオフィス文書に限定（PyPDF2/chardetで処理）
- ユーザー質問は主に英語；複雑さ判断ルールはキーワードと長さに基づく
- OpenSearch Serverlessベクトル検索によるTop‑K関連チャンクでQA要件を満たせる
- コスト見積もりはBedrockオンデマンド価格に基づく；実際との乖離は±20%以内

### 4.2 制約条件

- 個人実装のため、専任のデータサイエンスや運用チームは不在
- すべてのサービスは単一のAWSリージョンにデプロイ
- Lambda実行タイムアウト15分だが、ドキュメント処理は数秒以内で完了する見込み
- ベクトル化はTitan Embeddingsを使用し、他の埋め込みモデルには切り替えない
- サードパーティ製LLMや外部APIは導入しない

## 5. アーキテクチャ概要

提供されたドキュメントと図に基づき、システムは以下の3つのコアパイプラインで構成されます。

### 5.1 データ取り込みパイプライン（ドキュメント処理）

- **ドキュメントアップロード**：API Gateway + S3（生バケット）、プリサインドURLでセキュリティ確保
- **ワークフローオーケストレーション**：S3イベント → EventBridge → Step Functions（リトライ、ビジュアルデバッグ対応）
- **テキスト抽出**：Lambda（Python + PyPDF2/chardet）がドキュメント内容を解析
- **ベクトル化**：Bedrock Titan Embeddingsモデルを呼び出してベクトル生成
- **ベクトル保存**：OpenSearch Serverless（自動スケーリング、意味検索とメタデータフィルタリングをサポート）

### 5.2 クエリ・コスト認識パイプライン（中核）

- **ユーザー認証**：Amazon Cognitoユーザープール、JWT検証
- **クエリ入口**：API Gateway（REST）
- **中核オーケストレーション**：Orchestrator Lambda
  - 意味検索：OpenSearchが関連ドキュメントチャンクを返却
  - 複雑さ判断：ルールベース（質問の長さ、キーワード）
  - モデルルーティング：簡単な質問 → Titan Text G1‑Lite（低コスト）；複雑な質問 → Claude 3 Haiku/Sonnet（高能力）
  - コスト記録：DynamoDBに書き込み（モデル、トークン数、推定コスト）
- **応答**：回答 + 使用モデル + コスト見積もり + レイテンシ

### 5.3 コスト分析パイプライン

- **コストデータレイク**：DynamoDBを時間パーティションでS3にエクスポート（Parquet形式）
- **サーバーレスクエリ**：Amazon AthenaでSQL分析
- **可視化ダッシュボード**：Amazon QuickSight（総コスト傾向、モデル別シェア、単純/複雑質問比率、平均リクエストコスト）
- **予算アラート**：CloudWatchが日次累計コストを監視 → SNS通知（しきい値 $5/日）

## 7. リスク管理

| リスク                                                   | 影響 | 対策                                                         |
| -------------------------------------------------------- | ---- | ------------------------------------------------------------ |
| Bedrockモデル呼び出し遅延が大きく、ユーザー体験に影響    | 中   | API Gatewayのタイムアウトとリトライ設定；p99レイテンシ監視；持続する場合はより高速なモデルに切り替え |
| 複雑さ判断ルールの不正確さによる高コストモデルの過剰利用 | 中   | ヒューリスティックな初期ルール＋拡張パス（軽量分類器に置換可能）；定期的にコストログを分析してしきい値を調整 |
| ベクトル検索が無関係な内容を返し、回答品質が低下         | 中   | チャンキング戦略とメタデータフィルタリングを最適化；Top-Kを増やすか、後続拡張としてリランキングを導入 |
| OpenSearch Serverlessコスト超過                          | 低   | オンデマンド容量モードを使用；OCU使用量を監視；インデックス保持ポリシーを設定 |
| DynamoDBコストログの急成長                               | 低   | TTL自動クリーンアップ（例：90日）；長期保管はS3 Glacierにエクスポート |
| 未承認アクセスによる機密ドキュメント漏洩                 | 高   | Cognito強制認証 + API Gatewayオーソライザー + S3バケットポリシーで公開アクセス禁止；IAM最小権限 |

## 8. コスト管理

### 8.1 コスト構成

- **コンピューティングコスト**：Lambda呼び出し（ドキュメント処理、Orchestrator）、Step Functions状態遷移
- **AIモデルコスト**：Bedrock（Titan Embeddings、Titan Text、Claude 3） – 入力/出力トークン単位で課金
- **ストレージコスト**：S3（生ドキュメント＋処理済みデータ）、OpenSearch Serverless（ベクトルインデックス）、DynamoDB（ログ）
- **クエリコスト**：Athena – スキャンデータ量で課金
- **可視化**：QuickSightスタンダード版 – ユーザー単位（単一ユーザーは無料枠内）
- **その他**：API Gateway、CloudWatch、SNS（無料枠内）

### 8.2 最適化戦略

- 動的ルーティング：簡単な質問の約80%がTitan Liteにルーティングされ、コストはClaude 3の約1/8
- S3ライフサイクル：生ドキュメントは30日後にIA、90日後にGlacierに移行
- DynamoDB TTL：90日より古いログを自動削除
- OpenSearchオンデマンド容量 – アイドル時は課金なし
- AWS Budgetsアラートを設定、月間コストを$10以内に制御（デモ用途）

### 9.2 受入基準

- アップロード後30秒以内にドキュメントがベクトル化され検索可能
- クエリ応答時間の中央値 <1.5秒、p99 <3秒
- 簡単な質問がTitan Liteにルーティングされる割合 ≥90%、複雑な質問がClaude 3にルーティングされる割合 ≥95%
- 常にClaude 3を使用した場合と比較して総コスト削減率 ≥70%
- すべてのAPIリクエストは有効なJWTを必要とする
- コストダッシュボードが日次およびモデル別のコスト傾向を正しく表示する

## 10. 監視と運用

### 10.1 監視メトリクス

- Lambda：呼び出し回数、エラー率、実行時間、スロットリング回数
- Step Functions：実行失敗率、各ステップの所要時間
- OpenSearch：クエリレイテンシ、OCU使用量
- Bedrock：モデル呼び出し回数、トークン数、スロットリングエラー
- DynamoDB：書き込みスループット、ストレージサイズ
- カスタムコストメトリクス：`TotalDailyCost`、`CostPerModel` をCloudWatchに発行

### 10.2 ロギングとトレーシング

- すべてのLambdaログをCloudWatch Logsに出力
- Step FunctionsはオプションでX-Rayトレーシングを有効化
- 監査のためにCloudTrailログを定期的にS3にエクスポート

### 10.3 アラートルール

### 10.4 ディザスタリカバリ

- すべてのIaCテンプレートをGitでバージョン管理
- DynamoDBオンデマンドバックアップ（週次）

- 重要な設定（IAM、API Gateway）はバージョン管理下に置く

## 11. セキュリティポリシー

- **認証**：Amazon Cognitoユーザープール + API Gateway JWTオーソライザー
- **データ暗号化**：S3デフォルトSSE-S3、OpenSearchノード間暗号化、DynamoDBデフォルト暗号化
- **転送セキュリティ**：API GatewayはHTTPSを強制、すべてのサービス間呼び出しはTLSを使用
- **IAM最小権限**：
  
  1.Orchestrator Lambdaロール：OpenSearch読み取り、Bedrock呼び出し、DynamoDB書き込み、SNS発行
  
  2.ドキュメント処理Lambdaロール：生S3バケット読み取り、Bedrock Embeddings呼び出し、OpenSearch書き込み
  
  3.コストエクスポートロール：DynamoDB読み取り、S3書き込み、Athena呼び出し
- **ネットワーク分離**：OpenSearch Serverless VPCエンドポイント（オプション、必要に応じて）
- **入力検証**：API Gatewayリクエストサイズ制限、Lambda内でファイルタイプとサイズを検証
